## Pengantar

Dalam proses **Software Testing**, tiga elemen utama yang saling melengkapi adalah:

1. **Test Scenario** – Gambaran umum tentang apa yang akan diuji untuk memastikan fungsi aplikasi berjalan sesuai kebutuhan.  
2. **Test Case** – Rangkaian langkah detail pengujian, mencakup input, proses, dan hasil yang diharapkan.  
3. **Bug Report** – Laporan resmi ketika ditemukan kesalahan atau kegagalan sistem, berisi detail masalah, cara mereproduksi, serta hasil aktual yang muncul.

Ketiga komponen ini membantu tim penguji memastikan aplikasi bekerja sesuai harapan dan bebas dari bug sebelum dirilis.

---

## Perbedaan Test Scenario dan Test Case

| Elemen | Pertanyaan yang Dijawab | Tujuan |
|--------|--------------------------|--------|
| **Test Scenario** | “Apa yang harus diuji?” | Memberikan gambaran umum area pengujian |
| **Test Case** | “Bagaimana cara melakukan pengujian?” | Menjabarkan langkah-langkah spesifik dan hasil yang diharapkan |

---

## Template Test Scenario

| Field | Keterangan |
|-------|-------------|
| **ID Scenario** | Nomor unik skenario pengujian |
| **Deskripsi** | Ringkasan skenario pengujian |
| **Modul/Fitur** | Modul atau fitur yang diuji |

---

## Template Test Case

| Field | Keterangan |
|-------|-------------|
| **ID Test Case** | Nomor unik test case |
| **Deskripsi** | Ringkasan singkat pengujian |
| **Precondition** | Kondisi awal sebelum pengujian |
| **Test Steps** | Langkah-langkah pengujian |
| **Test Data** | Data yang digunakan |
| **Expected Result** | Hasil yang diharapkan |
| **Actual Result** | Hasil aktual dari pengujian |
| **Status** | Pass/Fail |

## Bug Report

### Pengertian
**Bug Report** adalah laporan formal mengenai kesalahan atau anomali dalam perangkat lunak.  
Laporan ini digunakan oleh tim pengembang untuk memperbaiki masalah dan memverifikasi perbaikan.

---

### Tingkatan Bug (Bug Severity)

| Severity | Dampak |
|-----------|--------|
| **Low** | Tidak berpengaruh pada sistem, hanya kosmetik |
| **Minor (Medium)** | Menyebabkan ketidaknyamanan kecil, fungsi utama tetap berjalan |
| **Major (High)** | Fungsi utama terganggu tapi sistem tidak crash |
| **Critical** | Sistem gagal total atau data rusak |

---

### Prioritas Bug (Bug Priority)

| Priority | Deskripsi |
|-----------|------------|
| **P1 – Urgent/Critical** | Harus segera diperbaiki karena menyebabkan sistem gagal |
| **P2 – High** | Berpengaruh besar pada pengguna, tapi tidak se-krusial P1 |
| **P3 – Medium** | Dapat diperbaiki pada rilis berikutnya |
| **P4 – Low** | Bug minor atau visual, bisa diperbaiki kapan saja |

---

### Contoh Format Bug Report

| Field | Keterangan |
|-------|-------------|
| **Bug Title** | Perhitungan BMI salah saat input berat 60kg dan tinggi 170cm |
| **Bug ID** | BMI-001 |
| **Step to Reproduce** | 1. Buka aplikasi BMI <br> 2. Masukkan Berat = 60 <br> 3. Masukkan Tinggi = 170 <br> 4. Klik tombol “Hitung” |
| **Actual Result** | Hasil BMI = 12.5 |
| **Expected Result** | Hasil BMI seharusnya = 20.8 |
| **Severity** | Major (High) |
| **Priority** | P2 – High |
| **Assignee** | Developer |
| **Reporter** | SQA (Software Quality Assurance) |
| **Reported On** | 31-08-2025 |
| **Build Number** | Version 1.0.0 |
| **Platform** | Android |

## Cara Menghindari Bug

1. **Pahami Persyaratan:** Semua anggota tim harus memahami requirement dengan jelas.  
2. **Unit Testing:** Lakukan pengujian unit sejak tahap awal pengembangan.  
3. **Code Review:** Minta developer lain meninjau kode untuk menemukan potensi kesalahan.  
4. **Rencana Pengujian:** Buat *test plan* yang menyeluruh dan terdokumentasi.  
5. **Automation Testing:** Gunakan alat otomatis untuk mempercepat deteksi bug.  
6. **Kolaborasi Tim:** Perkuat komunikasi antara developer dan tester agar proses QA lebih efektif.

---

## Kesimpulan

**Test Scenario**, **Test Case**, dan **Bug Report** adalah tiga pilar penting dalam kualitas perangkat lunak.  
Dengan membuat skenario pengujian yang jelas, menulis test case yang sistematis, dan melaporkan bug secara detail, tim QA dapat memastikan software lebih stabil, fungsional, dan siap digunakan oleh pengguna akhir.
